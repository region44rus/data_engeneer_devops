# Практическое задание

**Тема:** Совместная работа в GitHub: Issues, Branches, Pull Requests, Milestones, базовая Security

## Цель

* Освоить workflow через ветки и Pull Request
* Научиться формулировать и оформлять Issues
* Понять назначение Milestones
* Использовать GitHub Projects для отслеживания задач
* Ознакомиться с базовыми настройками безопасности репозитория
* Закрепить дисциплину логически структурированных изменений

## Задание (шаги)

Работа выполняется в ранее созданном репозитории `git-practice-<фамилия>`.

### 1. Создание и оформление Issues

1. Перейдите во вкладку **Issues**.

2. Создайте минимум **3 задачи**:

   **Issue 1:** Добавить расширенные заметки по Git
   **Issue 2:** Добавить раздел о Pull Requests
   **Issue 3:** Обновить структуру README

3. Для каждой задачи:

   * Добавьте осмысленное описание.
   * Назначьте себя (Assignee).
   * Добавьте Label (создайте при необходимости: `documentation`, `enhancement`).
   * Укажите Milestone (создадите его на следующем шаге).

### 2. Создание Milestone

1. Перейдите в раздел **Milestones** (справа).
2. Создайте Milestone:

   ```
   v1.1 - Workflow improvement
   ```
3. Привяжите все созданные Issues к этому Milestone.

### 3. Работа через ветки и Pull Request

Для каждой Issue:

1. Создайте отдельную ветку:

   ```
   feature/issue-1-notes-update
   feature/issue-2-pr-section
   feature/issue-3-readme-structure
   ```

2. Выполните изменения:

   * В `notes/git-basics.md` добавьте раздел:

     ```
     Что такое Pull Request?
     Зачем нужны ветки?
     ```
   * В `README.md` добавьте раздел:

     ```
     ## Работа через Pull Request
     ```
   * Улучшите структуру README (добавьте оглавление или более чёткие заголовки).

3. Сделайте коммит:

   * Сообщение должно начинаться с:

     ```
     feat: ...
     ```

     или

     ```
     docs: ...
     ```

4. Выполните Push ветки.

5. Создайте **Pull Request**:

   * В описании укажите:

     ```
     Closes #<номер_issue>
     ```
   * Добавьте описание изменений.
   * Проверьте вкладку “Files changed”.

6. Выполните Merge.

7. Убедитесь, что Issue автоматически закрылся.

### 4. Базовая Security-настройка

1. Перейдите во вкладку **Security**.
2. Проверьте:

   * Включён ли Dependabot alerts.
   * Включён ли Code scanning (если доступно).
3. Добавьте в README раздел:

   ```
   ## Security
   В проекте включены базовые механизмы защиты GitHub.
   ```

## Подсказки по ключевым частям

* **Issue = формализованная задача.**
* **Ветка = изолированная работа над задачей.**
* **Pull Request = механизм проверки изменений перед объединением.**
* Закрытие Issue через `Closes #номер` — профессиональный стандарт.
* Один PR — одна задача.
* Не работайте напрямую в `main`.

## Что проверить перед отправкой (чек-лист)

Перед сдачей убедитесь, что:

* [ ] Создано минимум 3 Issues
* [ ] Создан и использован Milestone
* [ ] Для каждой задачи создана отдельная ветка
* [ ] Создано минимум 3 Pull Request
* [ ] В PR используется `Closes #номер`
* [ ] Все PR смёржены
* [ ] Issues автоматически закрыты
* [ ] Проверены базовые Security-настройки
* [ ] В истории нет коммитов напрямую в main

## Советы по улучшению работы

* Используйте префиксы коммитов (`feat`, `fix`, `docs`).
* Делайте небольшие Pull Request.
* Проверяйте diff перед Merge.
* Пишите описание PR так, будто его читает другой разработчик.
* Используйте Milestones для планирования релизов.
* Поддерживайте порядок в Issues (закрывайте устаревшие).
